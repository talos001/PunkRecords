# F-2 动态领域与 Vault 路径策略设计

## 1. 背景与目标

`docs/function_plan.md` 中 F-2 议题要求领域从静态配置演进为可动态管理，且新增领域后可与 Vault 路径策略联动，不再依赖手工改 YAML 静态领域列表。

本设计聚焦完整前后端实现（用户确认范围 C），并落实两项关键决策：

- 删除采用受限策略（用户确认 C）：存在材料/索引数据时禁止删除，改为归档。
- 领域 ID 由后端根据名称自动生成 slug（用户确认 B），冲突自动追加后缀。

## 2. 范围与非范围

### 2.1 范围

- 后端提供动态领域管理接口：新增、查询、编辑、受限删除（归档优先）。
- 领域数据持久化替代静态 `DOMAINS` 直读，支持启动种子迁移。
- 前端提供领域管理 UI（新增、编辑、归档），并与现有领域选择联动。
- 领域路径解析采用统一优先级：`domain > user > global`。
- 与现有摄取管线（Vault/索引）路径解析逻辑对齐并统一入口。

### 2.2 非范围

- 组织级权限、多租户领域隔离。
- 第三方登录与认证机制变更（沿用 F-1）。
- 新增复杂权限角色（如“仅管理员可管理领域”）。
- 领域批量导入导出与审计报表。

## 3. 方案选择与推荐

对比三种候选：

- 方案 1：单一 `domains` 资源 + 受限删除校验中间件（推荐）
- 方案 2：静态默认领域 + 动态领域叠加层
- 方案 3：事件日志驱动（append-only）投影

选择方案 1，理由：

- 数据来源单一，前后端契约一致，减少“静态+动态双源”冲突。
- 满足当前业务需求同时保留后续扩展空间（恢复、审计、权限细化）。
- 与当前项目阶段复杂度匹配，不引入过重基础设施。

## 4. 架构与组件边界

### 4.1 后端分层

- `DomainRepository`：领域持久化读写（文件或后续存储后端）。
- `DomainService`：业务规则（slug 生成、重名冲突、受限删除、路径解析）。
- `DomainRouter`：HTTP 契约层（请求校验、鉴权、状态码映射）。

约束：路径解析和删除校验必须集中在 `DomainService`，避免散落在多个路由/调用点。

### 4.2 前端分层

- `DomainManagerPanel`：领域管理入口与列表操作。
- `DomainFormModal`：新增/编辑表单（`name/description/icon/paths`）。
- `useDomains()`：接口调用、缓存刷新、错误处理、乐观更新控制。

约束：聊天主界面的领域切换器仅承担“选择”职责，不叠加复杂管理行为。

### 4.3 迁移策略

- `GET /api/v1/domains` 成为领域列表权威来源。
- 启动时若持久层为空，执行一次默认领域种子写入（来源于现有静态定义）。
- 前端静态 `domains.ts` 降级为异常兜底，不再作为常规业务来源。

## 5. 数据模型与 API 契约

### 5.1 领域模型

- `id`：后端自动生成 slug，冲突后缀递增（`-2/-3/...`）。
- `name`：必填显示名，按归一化规则做唯一性校验。
- `description`：可选描述。
- `icon`：可选图标（emoji 或图标 key）。
- `status`：`active | archived`。
- `paths`：领域级路径配置（可选）。
- `created_at` / `updated_at`：时间戳。

说明：编辑领域名称时不回写 `id`，避免既有引用失效。

### 5.2 API 建议

- `POST /api/v1/domains`
  - 入参：`name/description/icon/paths?`
  - 出参：新建领域对象（含后端生成 `id`）
- `GET /api/v1/domains?include_archived=false`
  - 默认不返回归档领域
- `PATCH /api/v1/domains/{id}`
  - 可修改：`name/description/icon/paths/status`
- `DELETE /api/v1/domains/{id}`
  - 执行受限删除判定；存在数据时拒绝并引导归档

### 5.3 错误语义

- `400 DOMAIN_NAME_INVALID`
- `409 DOMAIN_NAME_CONFLICT`
- `409 DOMAIN_NOT_EMPTY`
- `404 DOMAIN_NOT_FOUND`
- `401 AUTH_REQUIRED`（沿用 F-1）

## 6. Vault 路径策略与数据流

### 6.1 路径解析优先级

统一规则：

1. 领域级 `paths`（若配置）
2. 用户级默认路径（F-1 的 `effective_materials_path`）
3. 系统全局默认路径

约束：调用方不得自行拼接路径，统一使用 `DomainService.resolve_paths(domain_id, user_id)`。

### 6.2 目录创建策略

- 新建领域成功后，不强制立即创建物理目录（lazy create）。
- 首次 ingest 或写入该领域时，如目录不存在再自动创建。
- 用户界面仅展示“当前生效保存位置”，不暴露实现层术语。

### 6.3 归档领域行为

- 归档领域禁止新写入。
- 历史数据保持可读。
- 领域恢复为 `active` 后继续按统一路径规则生效。

### 6.4 可观测性

在 ingest 结果或日志中记录：

- `effective_domain_id`
- `effective_paths_source`（`domain | user | global`）

用于快速排查路径来源与落盘行为。

## 7. 前端交互与异常恢复

### 7.1 新增/编辑流程

- 新增领域仅输入 `name/description/icon`（不暴露 `id` 输入）。
- 保存成功后再刷新列表并显示新领域。
- 名称冲突时保留表单内容并展示错误提示。

### 7.2 受限删除体验

- 以“归档”作为主文案与主路径。
- 若删除请求返回 `DOMAIN_NOT_EMPTY`，弹窗提示“已有材料，不能删除，可归档”。
- 用户确认后可一键执行归档，领域默认从主列表隐藏。

### 7.3 异常处理

- `401`：沿用 F-1 登录拦截。
- 网络失败：提示 + 保留用户输入可重试。
- 列表刷新失败：局部错误态，不影响聊天主流程。
- 任何失败不应清空已加载领域列表。

## 8. 测试与验收标准

### 8.1 后端测试

- slug 自动生成与冲突递增正确。
- 编辑领域不改变 `id`。
- `DOMAIN_NOT_EMPTY` 触发逻辑正确，归档路径可用。
- `include_archived` 过滤行为正确。

### 8.2 路径策略测试

- `domain > user > global` 优先级严格生效。
- 领域路径缺省回退正确，不产生空路径写入。
- 归档领域新写入被拒绝，历史读取可用。

### 8.3 前端测试

- 新增成功后列表可见并可被选择。
- 冲突错误可见且表单内容保留。
- `DOMAIN_NOT_EMPTY` 场景可引导归档并完成状态更新。
- 网络异常不造成列表闪空或状态错乱。

### 8.4 联调验收

- 登录后新增领域可在聊天领域选择器立即使用。
- 首次写入该领域时目录按规则自动解析并创建。
- 归档后默认不可选，可在“显示已归档”视图中查看与恢复。

## 9. 风险与缓解

- 风险：旧代码仍直读静态领域常量导致行为不一致。
  - 缓解：检索并替换为统一领域服务/接口读取。
- 风险：路径解析逻辑在多模块重复实现。
  - 缓解：强制统一走 `DomainService.resolve_paths`。
- 风险：前端删除语义与后端受限删除不一致引发误解。
  - 缓解：前端主文案改为“归档”，删除仅作为受限动作分支。
