## 2. 产品设计约束摘录（讨论议题时需对齐）

以下约束在讨论 §3 各议题时作为方向参考；若与 [`ui_design.md`](ui_design.md) 冲突，以 `ui_design` 及其中「维护规则」为准。

| 约束 | 说明 |
|------|------|
| **单一事实来源** | 用户可见文案、领域列表来源、Vault 路径等产品语义须有明确权威出处（如 `ui_design.md`、`api-outline.md`）；定稿并实施时须同步文档。 |
| **领域与 Vault** | 领域（`domain`）与材料/索引 Vault 的映射由服务端维护；用户侧不暴露实现层路径术语（见 `ui_design` §2、§3）。 |
| **配置分层** | 全局/环境配置（YAML、环境变量）与「每用户或每会话」配置区分清楚；引入登录后，**用户级配置**不得静默覆盖系统默认值而不经确认。 |
| **安全与密钥** | `llm_api_key` 等敏感信息不在前端明文长期存储于可被 XSS 读取的位置；优先服务端保管 + 会话态，或浏览器安全存储策略需单独评审。 |
| **Obsidian 兼容** | 材料落盘路径、图谱索引需继续满足现有 Vault 目录约定（见 `plan-b-vault-pipeline.md` 等），新设想不得破坏已有摄取管线假设。 |
| **班克记录人格** | 对话人格为毕达哥拉斯（Pythagoras），产品文案见 `frontend/src/brand.ts`；新界面避免使用泛称「助手」替代角色名（规范见 `ui_design` §6.4）。 |

## 4. 讨论议题说明

### 4.1 F-1：用户管理与材料 Vault 首次配置

**状态**：✅ 已完成（2026-04-13）

**讨论要点**

- UI 提供 **用户管理**（注册/登录/会话），与后端认证衔接（具体协议：Session / JWT / OIDC 等待定）。
- **登录后**，若当前用户尚未配置 **`materials_vault_path`**（或等价「材料库根路径」）：
  - **先展示配置引导**（说明用途、与 Obsidian 材料库关系）；
  - 若用户跳过或未填，则 **回退为默认**：使用 **全局默认配置文件**（如现有 `config.yaml` / `default_config`）中的 `materials_vault_path` 等参数，并在界面上 **明确展示当前生效路径**，要求用户 **确认** 后方可进入主功能（聊天、ingest、领域切换等）。
- **未登录**状态下，用户点击需身份的功能（聊天、设置、领域管理、摄取等）时，**统一提示先登录**，不提供半功能状态。

**若实施时需考虑**

- 后端：用户模型、配置存储（每用户一条或键值）、与现有 `Config` / `load_app_config` 的合并策略。
- 前端：路由守卫、登录态持久化、首登向导流程与 `ui_design` 对齐。

---

### 4.2 F-2：动态领域与 Vault 目录创建

**状态**：✅ 已完成（2026-04-13）

**讨论要点**

- UI 支持 **动态添加领域**（名称、描述、图标等按 `ui_design` 扩展）；**仅当添加成功（服务端持久化成功）后**，领域列表才显示新项。
- 每新增领域，用户可配置该领域对应的 Vault 目录结构（材料子路径、索引路径等按 `vaults` / `plan-b` 约定）；若无配置，则使用后台默认路径；**无需**手工改后台 YAML 中的静态领域列表。

**若实施时需考虑**

- 后端：`POST/DELETE /api/v1/domains`（或等价）与 `domains_data` 从静态改为持久化；与 `MaterialVault` / 索引路径生成逻辑一致。
- 迁移：现有硬编码 `DOMAINS` 与前端 `domains.ts` 需逐步改为接口驱动（与当前 `GET /domains` 演进一致）。

---

### 4.3 F-3：UI 配置模型参数

**讨论要点**

- 在 **设置**（或独立「模型」页）中支持用户配置：`llm_base_url`、`llm_api_key`、`llm_model`、`llm_provider`（如 `fake` / `anthropic`）。
- 保存策略：用户级优先；敏感字段展示与脱敏规则需符合 §2 约束。
- 2026-04-14 设计定稿：模型配置统一并入「设置」页 Tab，不再保留左侧独立模型入口；`llm_provider` 为可编辑下拉；采用手动保存。

**若实施时需考虑**

- 后端：扩展 `PUT /api/v1/settings` 或拆分 `PUT /settings/llm`；与 `LLMRegistry` 热更新或重启策略。
- 与 F-1 结合：未登录时不可改；登录后写入用户配置。

---

### 4.4 F-4：设置中修改 Vault 配置路径

**讨论要点**

- **设置页**支持修改 **`materials_vault_path`**（及后续若有的「默认领域索引根路径」等），保存后生效规则与 F-1 一致（校验路径可写、存在性提示）。
- 2026-04-14 设计定稿：路径配置与领域管理合并为同一 Tab；支持全局路径 + 每领域路径覆盖；覆盖留空时继承全局路径。

**若实施时需考虑**

- 与 F-1 重叠部分：首次引导与设置页编辑应共用同一套校验与持久化 API。
- 变更路径后的 ingest / 聊天落盘路径一致性需在实现单测或手册中说明。

---

### 4.5 URL 抓取与 ingest 串联

**状态**：讨论中

**讨论要点**

- 用户消息中的 URL 由后端抓取、转 Markdown/文本、落盘到材料 Vault，再进入摄取管线（见 `api-outline` §3.1）。
- 与 Plan B 阶段 3「URL 抓取」对齐，避免前端单独处理抓取与清洗流程。

**若实施时需考虑**

- 后端：HTTP 客户端超时、重试、速率限制与失败兜底策略。
- 可观测性：抓取失败原因与 ingest 失败原因需区分记录，便于排障。

---

### 4.6 异步任务（Plan C）

**状态**：讨论中

**讨论要点**

- 慢 ingest / 建图场景下，接口返回非空 `job_ids`，前端通过任务状态接口轮询。
- 提供 `GET /api/v1/jobs/{job_id}`，状态至少覆盖 `pending | running | success | failed`，并返回简要日志。

**若实施时需考虑**

- 任务生命周期与过期清理策略（避免状态表无限增长）。
- 前端轮询节流策略与失败重试策略。

---

### 4.7 会话与历史（可选）

**状态**：讨论中

**讨论要点**

- 提供 `POST/GET /api/v1/conversations`、`GET/POST .../messages`（见 `api-outline` 第 4 节）以支持会话历史持久化。

**若实施时需考虑**

- 与当前登录态绑定的数据隔离（不同用户会话不可串读）。
- 历史消息分页与加载性能（避免单会话过大导致接口超时）。

---

### 4.8 图谱 / Obsidian 入口

**状态**：讨论中

**讨论要点**

- 提供 `GET /api/v1/graph/open?domain_id=...`，返回自定义 URI 或 Obsidian 插件深链（见 `api-outline` 第 8 节）。

**若实施时需考虑**

- 深链协议在不同平台（macOS/Linux/Windows）的一致性与回退行为。
- 未安装插件或目标 Vault 不可用时的用户提示策略。

---

### 4.9 模型侧能力深化

**状态**：讨论中

**讨论要点**

- `BaseAgent.ingest` 当前多为占位空实体；需接入 LLM/规则抽取后写入 `graph_index`，才能产生业务价值。
- CLI `query` / `lint` 仍待补齐，需与 API 能力边界保持一致。

**若实施时需考虑**

- 抽取质量评估标准（精确率/召回率或最小可用规则集）。
- CLI 与 API 在模型选择、超时、错误码上的一致性。

---

### 4.10 前端页面效果优化

**状态**：讨论中

**讨论要点**

- 在不改变核心信息架构的前提下，优化页面视觉层次、留白、对齐与组件一致性，提升整体观感与可读性。
- 聚焦高频场景（聊天主区、侧栏、设置面板、领域管理）做细节打磨：状态反馈、悬停/禁用态、按钮层级与空态表现。

**若实施时需考虑**

- 与 `docs/ui_design.md` 保持单一事实来源，所有视觉与交互改动需同步更新该文档。
- 兼顾性能与可维护性：优先复用现有样式体系，避免大规模样式散落与重复定义。
