# PunkRecords UI 设计规范

本文档是 **PunkRecords 客户端/UI 层** 的产品与视觉设计的唯一事实来源（single source of truth）。  
**所有与 UI 相关的决策、迭代与讨论结论，都必须写入并更新本文件**，避免散落在聊天或零散 PR 描述中。

---

## 文档维护规则（必须遵守）

1. **新增**：任何新的界面能力、交互流程、文案原则、默认行为，先在本文档中补充或修订，再实现代码（紧急热修复可在合并后补记，但不得长期滞后）。
2. **变更**：交互或视觉有重大调整时，更新对应章节，并在文末 **修订记录** 中追加一行（日期、摘要、可选关联 PR/issue）。
3. **术语**：用户可见文案禁止使用实现层术语（如 Vault、ingest、query）；若必须在文档中说明与后端对应关系，放在「系统侧」小节，并标明「不对用户展示」。
4. **引用**：`README.md`、`CLAUDE.md` 仅作指向本文件的入口，**不重复**全文规范；详细内容以本文件为准。

## 1. 与架构的关系（`arch/PunkRecords_Arch.md`）

UI 层在架构中承担：

| 能力 | 说明 |
|------|------|
| 与用户交互 | 以对话为主壳，承载提问与指令 |
| 材料入口 | 文件、链接等形式进入系统（后端再完成解析与入库） |
| 领域上下文 | 用户选择「知识区域」，后端映射到具体 Vault |
| 图谱查看 | 全量关系图谱以 **Obsidian 插件** 为主；主应用可提供入口、元数据与深链，不替代插件内编辑与浏览体验 |

系统侧仍存在 **LLM 代理层**、**Obsidian Vaults 层**、ingest/query/lint、graphify 等；**这些名词不对用户暴露**。

## 2. 用户心智：领域（可见）与 Vault（不可见）

### 2.1 原则

- 用户只接触 **知识区域 / 领域**（如：幼儿发展、数学、英语、语文、历史等），用名称、图标、简短说明呈现。
- **Vault 路径、索引库、原材料库** 仅存在于配置与后端映射，**永不出现在 UI 文案中**。

### 2.2 后端映射（概念）

系统维护注册表，例如：`domain_id → { material_vault, index_vault(s), … }`（具体字段以实现为准）。  
UI 只提交 `domain_id`（或同等标识），由服务端解析 Vault。

### 2.3 未归类 / 扩展

若未来需要「临时堆放」，可增设用户可见名称如 **收件箱**；是否允许多领域引用同一份材料，由产品另定，确定后写入本文档并更新交互说明。

---

## 3. 无「ingest / query / link」术语的产品表达

用户不选择「ingest」或「query」，只提供 **材料** 与 **自然语言意图**。

| 用户行为 | 界面呈现（示例） | 系统侧（不对用户展示） |
|----------|------------------|-------------------------|
| 文件（见下节格式） | 在**对话输入框**内粘贴文件或拖入文件，与文字一并发送 | 解析 → 写入对应存储 → 更新索引管线 |
| 网页 URL | 在输入框内粘贴链接，与文字一并发送 | 拉取/解析 → 同上 |
| 纯文字对话 | 在输入框内输入并发送 | 在当前领域下检索与回答 |

**推荐用户向用词**：添加、导入、整理、问一问、从网页保存、总结、帮我看看……  
**禁用用户向用词**：ingest、query、lint、Vault、索引库、原材料库（除非内部调试模式且默认关闭）。

**隐式意图**：有附件/链接时优先按「材料进入当前领域」处理；纯文本优先按「问答」。若歧义大，用 **一句澄清**（仍用自然语言），不暴露技术管线名称。

## 4. 支持的材料类型（当前范围）

- **Markdown**（`.md`）
- **PDF**（`.pdf`）
- **Web URL**（用户粘贴；由系统拉取并解析）

其他格式待支持时，须在本节增补并说明错误提示策略。

## 5. 交互顺序：先领域，再操作；默认选中领域

### 5.1 规则

1. **进入应用时** 必须已有一个 **当前选中的领域**（见下「默认策略」）。
2. **上传文件、粘贴链接、发送消息** 均在 **当前领域** 下执行；用户无需在每次操作前再点「确认领域」（因默认已选中）。
3. **切换领域** 通过 **知识区域卡片**（置于聊天区上方，非下拉列表）。切换后，**清空当前对话线程**（含未发送的输入草稿与待附文件），回到无消息时的空状态；**后续** 发送内容归属新领域；进行中的流式回复会被中止。可选轻提示当前区域说明。

### 5.2 默认领域策略

- **默认领域**：**幼儿发展**（`early-childhood`）；不提供名为「通用」的领域。
- **首次使用**：默认选中上述领域。
- **再次打开**：优先恢复 **上次选中的领域**（建议本地持久化）；无记录或旧版「通用」遗留数据时，回退到 **幼儿发展**。

### 5.3 与「先选领域再上传」的关系

「先选领域」落实为：**任一写入类操作发生时，领域已确定**——通过 **默认选中** 与 **可切换**，避免用户每次从零选择；仅当无默认且必须让用户明确时，再强制选择。

## 6. 信息架构（IA）建议

### 6.1 应用壳（App Shell）

1. **布局（桌面）**：**左右分栏**——**左侧**为 **可折叠侧栏**（默认展开；折叠后仅保留窄轨图标 + 小 Logo）；**右侧**为主工作区（顶栏含侧栏开关 + 标题 + 领域 + 对话 + 输入）。折叠状态可持久化（`localStorage`）。
2. **侧栏导航（左侧，可扩展）**：除品牌外，提供 **主页**、**对话风格**、**设置** 等入口（当前为占位，选中态高亮）；折叠时仅显示图标，`title` 提供完整名称。
3. **品牌区（侧栏底部）**：**Logo**（`frontend/public/punkrecords_logo.png`）、主标题 **「PunkRecords · 班克记录」**、副文案与 README 一致，见 `frontend/src/brand.ts` 中 **`PRODUCT_TAGLINE`**。展开时完整展示；折叠时仅保留小 Logo。侧栏背景为浅灰以区分主区白底。
4. **领域选择（右侧、输入框上方）**：知识区域为 **横向药丸按钮**（emoji + 名称），单选；**不使用下拉框**；选中态为 **橙色描边 + 浅橙底**（简洁白底风格）。
5. **对话区**：会话线程；**一行**当前领域说明（居中、弱对比文字即可）。**尚无消息时**，将「主标题 + 领域药丸 + 当前区域说明 + 空状态提示 + 输入区」作为整体在主区 **垂直居中**；产生消息后改为常规 **自顶向下** 排布，消息列表区域可滚动。
6. **输入区**：**一体化圆角容器**——左侧 **回形针**（唤起文件选择，仍无独立「上传」表单行）、中间 **多行输入**、右侧 **圆形发送**；支持粘贴文件、拖入文件与纯文字；解析由后端完成。
7. **侧栏（可选，未来）**：「最近添加」「最近打开的网页」等——用 **标题/时间** 展示，避免技术日志式列表。
8. **可选**：在输入区上方用一行轻提示列出「将随下一条消息发送」的文件名（来自粘贴/拖入）。
9. **窄屏**：保持左右分栏；侧栏展开宽度在窄屏上受限（如 `min(260px, 78vw)`）；药丸可换行。

### 6.2 图谱（Obsidian）

- 主 UI：**打开图谱 / 在 Obsidian 中查看**（深链或本地协议），可展示少量统计（如上次构建时间）；**不**承担完整图谱编辑职责。
- 详细可视化以 **Obsidian 插件** 为准（与架构图一致）。

### 6.3 写库安全

- 写入或覆盖前，尽量提供 **路径或标题级预览**（用户能理解的形式），避免误写。

### 6.4 对话角色（用户可见）

- **名称**：**毕达哥拉斯（Pythagoras）**；设定为 **贝加庞克的智慧分身**，智慧的代表，负责 **知识存储与数据分析**。
- **文案**：用户可见处 **不使用「助手」**；气泡抬头与加载态见 `frontend/src/brand.ts`（`PYTHAGORAS_NAME` / `PYTHAGORAS_THINKING`）。
- **后端 system prompt**：见 `src/api/chat_profiles.py`，与上述设定一致。

## 7. 视觉与气质（方向性约束）

以下为方向约束，具体组件规格可在实现阶段细化，但 **须与本节气质一致**，避免「通用 AI 界面」同质化。

- **气质**：简洁、留白充足、偏「工具型」白底界面；可与品牌「班克记录」呼应；**可读性优先**（长对话、长文）。**避免**整屏纯黑或 heavy 渐变。
- **字体**：当前 Web 使用 **Inter**（全站），见 `frontend/index.html`。
- **色彩**：**白 / 浅灰** 为主（主内容区 `#fff`，页背景约 `#fafafa`，侧栏约 `#f5f5f7`）；**细边框** 与轻阴影；主操作 **橙**（与参考图一致的量级，可微调色值）。
- **动效**：首屏可有克制的 stagger 入场；任务完成用轻量反馈即可，避免喧宾夺主。
- **版面**：鼓励一定 **非对称或层级对比**，避免千篇一律的居中对话卡片堆叠。

（若后续输出 Design Token，可追加「设计令牌」子节。）

## 8. 风险与对策（摘要）

| 风险 | 对策 |
|------|------|
| 选错领域 | 显著展示当前领域；发送前可选确认；支持短时间内「改存到其他领域」（文案仍用领域名） |
| URL/PDF 失败 | 用户可见：无法打开链接 / 无法读取文件；详情进日志 |
| 多领域意图 | 用户明确说「放到数学里」时，通过 **切换领域** 或 **@领域**（若产品启用）解决，规则确定后补写本节 |

## 9. 技术实现与仓库布局

### 9.1 语言与栈（已采纳）

| 层级 | 选型 | 说明 |
|------|------|------|
| 用户界面 | **TypeScript + React 18** | 类型安全、组件化，便于聊天、附件与后续 Markdown 预览 |
| 构建 | **Vite 5** | 开发热更新快，生产构建简单 |
| 样式 | **CSS**（变量 + 组件级 `App.css`） | 无额外 UI 框架依赖，便于落实本文档的视觉方向 |
| 后端（现有） | **Python**（`src` 包） | 领域映射、Vault、Agent 等；与 UI 通过 **HTTP API** 对接（待接） |
| 桌面壳（可选、未做） | **Tauri** | 可将同一 `frontend` 打包为桌面应用，需单独里程碑 |

不推荐用 Python 桌面框架（如 PyQt）做主聊天界面：开发与社区生态对现代 Web 交互支持较弱；**Python 保留为服务端与 CLI** 更符合当前架构。

### 9.2 代码位置

- `frontend/`：Vite 项目，`npm run dev` 本地预览，`npm run build` 产出静态资源至 `frontend/dist/`。
- 领域列表与默认领域：`frontend/src/domains.ts`（与后端 `domain_id` 对齐时需同步配置或改为接口拉取）。
- 侧栏导航项（占位）：`frontend/src/sidebarNav.ts`。
- 侧栏折叠状态：`localStorage` 键 `punkrecords_sidebar_collapsed`（`1` / `0`）。
- Logo：源文件可参考 `docs/punkrecords_logo.png`；构建时使用 `frontend/public/punkrecords_logo.png`（由 Vite 以根路径 `/punkrecords_logo.png` 提供）。

### 9.3 后续对接

- 聊天与上传应调用后端 API（如 FastAPI），请求中携带 **当前 `domain_id`**，由服务端解析 Vault；**不在前端暴露路径**。
- 接口路径与字段的完整清单见 **[api-outline.md](api-outline.md)**（随前后端约定迭代更新）。
